# Internacionalización (i18n) — 365Soft Frontend

Sistema de traducciones basado en **Transloco v8** (`@jsverse/transloco`).  
Soporta **español (es)** e **inglés (en)**. Arquitectura de scopes por módulo para escalar a 2000+ claves sin degradar rendimiento.

## 1. Arquitectura general

```
Browser solicita la app
        ↓
Transloco carga es.json (global — common, sidebar, auth)
        ↓
Usuario navega a /contratos
        ↓
Transloco carga es/contratos.json (solo ese módulo)
        ↓
Usuario cambia idioma → carga en.json + en/contratos.json
```

**El usuario nunca descarga traducciones de módulos que no visitó.**

## 2. Estructura de archivos

```
public/i18n/
├── es.json                    ← Claves globales (common, sidebar, auth)
├── en.json                    ← Claves globales en inglés
│
├── es/                        ← Scopes en español
│   ├── portal-publico.json    ← public.*
│   ├── landing.json           ← landing.*
│   ├── rentalApp.json         ← rentalApp.* (wizard de solicitud de alquiler)
│   ├── dashboard.json         ← dashboard.*
│   ├── propiedades.json       ← properties.* + propertyTypes.*
│   ├── contratos.json         ← contracts.* + tenants.*
│   ├── solicitudes.json       ← solicitudes.*
│   ├── pagos.json             ← payments.*
│   ├── mantenimiento.json     ← maintenance.* + tecnico.*
│   ├── empleados.json         ← employees.*
│   ├── notificaciones.json    ← notifications.*
│   ├── configuracion.json     ← config.* + wizard.*
│   └── portal-inquilino.json  ← tenantContracts.* + tenantDocuments.*
│                                 tenantApplications.* + perfil.*
└── en/
    └── (mismos 13 archivos en inglés)
```

### ¿Qué va en el JSON global vs en un scope?

| Global `es.json` | Scope `es/<modulo>.json` |
|---|---|
| Botones genéricos (`common.save`, `common.cancel`) | Texto específico de un módulo |
| Mensajes de error compartidos (`common.errors.*`) | Labels de formularios de ese módulo |
| Sidebar y navegación | Títulos y descripciones de ese módulo |
| Flujo de autenticación (`auth.*`) | Confirmaciones y diálogos de ese módulo |

**Regla:** Si el texto solo aparece en un módulo → va en el scope. Si aparece en 3+ módulos → va en `common.*`.

## 3. Cómo agregar una traducción nueva

### Paso 1 — Agregar la clave en el JSON del módulo correspondiente

```jsonc
// public/i18n/es/contratos.json
{
  "contracts": {
    "title": "Contratos",
    "newBtn": "Nuevo Contrato",
    "myNewKey": "Mi nuevo texto"   // ← agregar aquí
  }
}
```

### Paso 2 — Agregar la misma clave en inglés

```jsonc
// public/i18n/en/contratos.json
{
  "contracts": {
    "title": "Contracts",
    "newBtn": "New Contract",
    "myNewKey": "My new text"   // ← misma clave, traducida
  }
}
```

### Paso 3 — Usar en el template o en TypeScript

```html
{{ 'contracts.myNewKey' | transloco }}
```

> **Importante:** `es.json` y `en.json` siempre deben tener exactamente las mismas claves. Si falta una clave en algún idioma, Transloco cae al idioma de fallback (`es`) en lugar de mostrar la clave en crudo.

## 4. Cómo agregar un módulo nuevo

Ejemplo: se agrega el módulo `Propietarios`.

### 1. Crear los archivos de scope

```bash
touch public/i18n/es/propietarios.json
touch public/i18n/en/propietarios.json
```

```jsonc
// public/i18n/es/propietarios.json
{
  "owners": {
    "title": "Propietarios",
    "subtitle": "Gestión de propietarios",
    "addBtn": "Agregar Propietario"
  }
}
```

```jsonc
// public/i18n/en/propietarios.json
{
  "owners": {
    "title": "Owners",
    "subtitle": "Owner management",
    "addBtn": "Add Owner"
  }
}
```

### 2. Declarar el scope en los componentes del módulo

```typescript
import { provideTranslocoScope } from '@jsverse/transloco';

@Component({
  selector: 'app-propietarios',
  standalone: true,
  providers: [provideTranslocoScope('propietarios')],  // ← nombre del archivo sin extensión
  // ...
})
export class PropietariosComponent {}
```

### 3. Usar las claves normalmente

```html
<h1>{{ 'owners.title' | transloco }}</h1>
```

## 5. Uso en templates

### Pipe básico

```html
<h1>{{ 'contracts.title' | transloco }}</h1>
```

### Con interpolación de variables

```html
<!-- JSON: "welcome": "Bienvenido, {{name}}" -->
<p>{{ 'auth.welcome' | transloco: { name: userName() } }}</p>
```

### Con directiva estructural (bloque completo)

```html
<ng-container *transloco="let t">
  <h1>{{ t('contracts.title') }}</h1>
  <p>{{ t('contracts.subtitle') }}</p>
</ng-container>
```

### En atributos HTML

```html
<input [placeholder]="'contracts.searchPlaceholder' | transloco" />
<button [attr.aria-label]="'common.delete' | transloco">
  <lucide-icon [img]="TrashIcon" />
</button>
```

## 6. Uso en TypeScript

Inyectar `TranslocoService` con `inject()`:

```typescript
import { TranslocoService } from '@jsverse/transloco';

@Injectable({ providedIn: 'root' })
export class ContractService {
  private transloco = inject(TranslocoService);

  loadContracts(): void {
    this.http.get<Contract[]>(endpoint).pipe(
      catchError(() => {
        // Traducción en tiempo de ejecución
        this.errorSignal.set(this.transloco.translate('common.errors.loadContracts'));
        return of([]);
      }),
    ).subscribe();
  }
}
```

> **Nota:** En servicios con `providedIn: 'root'`, `TranslocoService` está disponible directamente. No hace falta `provideTranslocoScope` en servicios — solo en componentes.

## 7. Scopes — carga lazy por módulo

### ¿Qué es un scope?

Un scope es un archivo JSON de traducciones que se carga **solo cuando se renderiza un componente que lo declara**. Mientras el usuario no visite ese módulo, el archivo no se descarga.

### Cómo funciona el loader

```typescript
// src/app/app.config.ts
class TranslocoHttpLoader implements TranslocoLoader {
  private http = inject(HttpClient);

  getTranslation(lang: string, options?: { scope?: string }): Observable<Translation> {
    const scope = options?.scope;
    const path = scope ? `/i18n/${lang}/${scope}.json` : `/i18n/${lang}.json`;
    return this.http.get<Translation>(path);
  }
}
```

- Sin scope → carga `/i18n/es.json` (global)
- Con scope `contratos` → carga `/i18n/es/contratos.json`

### Declarar un scope en un componente

```typescript
@Component({
  providers: [provideTranslocoScope('contratos')],
})
export class ContractsComponent {}
```

## 8. Interpolación y pluralización

### Variables simples

```jsonc
// es.json
"daysAgo": "Hace {{days}} días"
```

```html
{{ 'common.daysAgo' | transloco: { days: 5 } }}
<!-- Resultado: "Hace 5 días" -->
```

### Variables dinámicas en servicios

```typescript
this.transloco.translate('common.errors.orgNotFound', { slug: 'mi-empresa' });
// Resultado: 'La organización "mi-empresa" no existe'
```

### Pluralización (con `@jsverse/transloco-messageformat`)

```jsonc
"items": "{count, plural, =0 {Sin elementos} =1 {1 elemento} other {# elementos}}"
```

```html
{{ 'contracts.items' | transloco: { count: total() } }}
```

> La pluralización avanzada requiere instalar `@jsverse/transloco-messageformat`. Para el proyecto actual, la mayoría de casos se resuelven con condicionales `@if` en el template.

## 9. Cambio de idioma en runtime

```typescript
import { TranslocoService } from '@jsverse/transloco';

@Component({ ... })
export class LanguageSelectorComponent {
  private transloco = inject(TranslocoService);

  currentLang = this.transloco.getActiveLang();

  switchLang(lang: 'es' | 'en'): void {
    this.transloco.setActiveLang(lang);
    localStorage.setItem('lang', lang);
  }
}
```

Al cambiar de idioma, Transloco recarga automáticamente todos los archivos activos (global + scopes en uso) y re-renderiza los templates sin recargar la página.

## 10. Reglas del equipo

| Regla | Motivo |
|---|---|
| Toda clave debe existir en **ambos** idiomas (`es` y `en`) | Evitar que el fallback muestre claves en crudo |
| Usar **namespacing** estricto (`contracts.title`, no solo `title`) | Evitar colisiones entre módulos |
| Texto que aparece en **3+ módulos** va en `common.*` | DRY — no duplicar traducciones |
| **Nunca** hardcodear texto español en templates o TypeScript | Cualquier texto visible al usuario debe estar en los JSON |
| Mensajes de error de servicios usan `common.errors.*` | Centralización — un solo lugar para editar |
| Al agregar un scope nuevo, documenta la tabla del [punto 7](#7-scopes--carga-lazy-por-módulo) | Mantener este documento actualizado |

## 11. Preguntas frecuentes

**¿Por qué no un solo `es.json` con todas las claves?**  
Con 2000+ claves en un solo archivo, el mantenimiento se vuelve inmanejable y el usuario descarga traducciones de módulos que nunca visita. Los scopes resuelven ambos problemas.

**¿Es seguro? ¿Pueden atacar los JSON?**  
Los archivos son assets públicos estáticos, como un `.css`. No hay superficie de ataque. Angular escapa todo output en templates por defecto (protección XSS incluida).

**¿Va a ser lento cargar tantos archivos?**  
No. Cada archivo pesa entre 3-7 KB sin comprimir (~1-2 KB con gzip). Se cargan una sola vez y se cachean. Después cada traducción es un lookup en un objeto JavaScript en memoria (~0.001ms).

**¿Qué pasa si una clave no existe?**  
Transloco usa `fallbackLang: 'es'` — si la clave no existe en `en`, muestra el español. Si tampoco existe en `es`, muestra la clave en crudo (ej: `contracts.title`). El `missingHandler` está configurado para loggear en consola en desarrollo.

**¿Tengo que agregar `provideTranslocoScope` a todos los componentes hijos?**  
No. Solo al componente raíz de cada módulo (el que está en el router). Los componentes hijos que se renderizan dentro de ese árbol heredan el scope automáticamente via DI de Angular. Los componentes que tienen su **propia ruta lazy** sí necesitan declarar el scope individualmente.

**¿Cómo sé si olvidé una traducción?**  
Instalar `@jsverse/transloco-keys-manager` y ejecutar `npx transloco-keys-manager find`. Detecta claves usadas en el código que no existen en los JSON, y claves en los JSON que ningún componente usa.
